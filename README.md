# Проект: "Кэширующий прокси с инвалидацией" (Caching Proxy)

## Суть проекта

HTTP-сервер, который работает как reverse-proxy для любого внешнего API (например, JSONPlaceholder или GitHub API), кэширует ответы в Redis и умеет инвалидировать кэш по времени и через "сброс" (invalidation endpoint).

## Почему именно он?

1. **Настоящий production-сценарий** — микросервисы часто кэшируют ответы внешних API
2. **Работа с внешними зависимостями** — нужно поднять Redis, настроить подключение
3. **Паттерны конкурентности** — защита от "стадного эффекта" (thundering herd)
4. **HTTP-сервер middlewares** — логирование, метрики, таймауты
5. **Конфигурация через окружение** — 12-factor app подход
6. **Graceful shutdown** — корректное завершение

## Техническое задание

### Функциональные требования

1. **Проксирование запросов**
   - Сервер принимает запросы вида `/api/*` и проксирует их на целевой API
   - Например: `GET /api/posts/1` → `https://jsonplaceholder.typicode.com/posts/1`
   - Поддержка методов: GET (для кэширования), остальные методы просто проксируются

2. **Кэширование GET-запросов**
   - Ответы на GET-запросы сохраняются в Redis
   - Ключ кэша: метод + URL + query-параметры (нормализованные)
   - TTL для кэша задается через конфигурацию (по умолчанию 5 минут)
   - Кэшируются только успешные ответы (статусы 2xx)

3. **Защита от "стадного эффекта"**
   - Если 10 concurrent запросов приходят на один и тот же некэшированный ресурс, только один идет в целевой API, остальные ждут его результат
   - Использовать singleflight или реализовать свой механизм через sync.Map + каналы

4. **Инвалидация кэша**
   - Endpoint `POST /admin/cache/invalidate` с авторизацией по простому токену (в хедере X-API-Key)
   - Тело запроса: `{"pattern": "/api/posts/*"}` — удаляет все ключи, подходящие под паттерн
   - Использовать Redis `SCAN` для поиска ключей по паттерну

5. **Метрики и мониторинг**
   - Эндпоинт `/metrics` для Prometheus (можно использовать готовую библиотеку)
   - Собирать: количество запросов, hit/miss ratio, время ответа, размер кэша (приблизительно)

### Технические требования

**Стек:**

- Go 1.25+
- Redis (через docker-compose)
- Библиотеки (можно выбрать):
  - `github.com/go-redis/redis/v8` — работа с Redis
  - `github.com/gorilla/mux` или `net/http` для роутинга
  - `github.com/prometheus/client_golang` для метрик
  - `golang.org/x/sync/singleflight` для защиты от стадного эффекта

**Конфигурация через переменные окружения:**

```env
TARGET_API=https://jsonplaceholder.typicode.com
REDIS_ADDR=localhost:6379
REDIS_PASSWORD=
REDIS_DB=0
CACHE_TTL=5m
SERVER_PORT=8080
ADMIN_TOKEN=secret-token-123
```

### Что нужно реализовать (по шагам)

**Шаг 1. Базовое прокси (1 час)**

- Загрузка конфигурации из env
- HTTP-сервер с одним обработчиком, который проксирует запрос на TARGET_API
- Поддержка всех методов (просто копируем тело и заголовки)
- Graceful shutdown

**Шаг 2. Кэширование GET-запросов (1 час)**

- Поднятие Redis через docker-compose
- Реализация интерфейса кэша (Get, Set, Delete)
- При GET-запросе: проверяем кэш, если есть — возвращаем, если нет — делаем запрос и сохраняем
- Сохраняем статус код, заголовки (Content-Type) и тело

**Шаг 3. Защита от стадного эффекта (30 минут)**

- Интеграция singleflight
- При множестве одновременных запросов к одному ключу — только один идет в API

**Шаг 4. Инвалидация кэша (1 час)**

- Admin middleware с проверкой токена
- Эндпоинт для инвалидации по паттерну
- Использование SCAN для удаления (не KEYS!)

**Шаг 5. Метрики (30 минут)**

- Подключение Prometheus
- Счетчики запросов (всего, по статусам)
- Гистограмма времени ответа
- Счетчик cache hits/misses

### Проверка работы

Запуск инфраструктуры:

```bash
docker-compose up -d redis
```

Запуск прокси:

```bash
export TARGET_API=https://jsonplaceholder.typicode.com
export REDIS_ADDR=localhost:6379
export ADMIN_TOKEN=secret
go run cmd/proxy/main.go
```

Тесты через curl:

```bash
# Первый запрос (идет в API, кэшируется)
curl -v http://localhost:8080/api/posts/1

# Второй запрос (должен прийти из кэша) - смотрим по заголовку X-Cache: HIT
curl -v http://localhost:8080/api/posts/1

# Инвалидация кэша
curl -X POST http://localhost:8080/admin/cache/invalidate \
  -H "X-API-Key: secret" \
  -H "Content-Type: application/json" \
  -d '{"pattern":"/api/posts/*"}'

# Проверяем метрики
curl http://localhost:8080/metrics
```

### Критерии "сдачи"

Проект считается успешно выполненным, если:

1. Кэш работает: повторные запросы возвращаются мгновенно с заголовком X-Cache: HIT
2. Singleflight работает: при одновременном запуске 10 curl'ов в background только один уходит в API
3. Инвалидация работает: после вызова /admin/cache/invalidate следующий запрос снова идет в API
4. Метрики показывают реальную картину
5. Код покрыт комментариями и следует идиомам Go
6. Нет утечек горутин и соединений с Redis
7. Graceful shutdown работает (обработка SIGTERM)
